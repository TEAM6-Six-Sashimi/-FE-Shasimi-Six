# 웹소켓 인증 토큰 노출 문제 — 백엔드 작업 요청 문서

이 문서는 프론트엔드에서 이미 개발/배포된 커피챗 웹소켓 기능에서 발견된 보안 이슈를 정리한 것입니다. 이 작업을 처음 맡는 사람(또는 AI)이 아래 내용만 보고 배경과 할 일을 파악할 수 있도록 작성했습니다.

---

## 0. 배경 — 지금 뭐가 이미 만들어져 있는지

프론트엔드에 "커피챗"이라는 기능이 있습니다. 학생이 수강 중인 강의의 강사와 실시간 채팅을 하는 기능이고, 아래처럼 이미 구현되어 배포까지 됐습니다.

- 채팅 실시간 송수신은 **WebSocket + STOMP** 프로토콜로 처리 (`ws://{host}/ws-coffeechat`, STOMP CONNECT 프레임의 `Authorization` 헤더에 JWT를 실어서 인증)
- 이 부분은 브라우저에서 직접 WebSocket 연결을 맺어야 해서, 브라우저 JS가 로그인 세션 토큰(`accessToken`)값을 알아야 합니다
- 그런데 `accessToken`은 `httpOnly` 쿠키라서 브라우저 JS가 원래 이 값을 못 읽습니다 (XSS 방지 목적으로 이렇게 설계되어 있었음)
- 이 문제를 풀기 위해, 프론트엔드에 `GET /api/ws-token`이라는 자체 API 라우트를 만들어서, **서버사이드에서 httpOnly 쿠키를 읽어 그 값을 그대로 JSON으로 브라우저에 내려주는 방식**으로 우회했습니다

이 방식이 지금 **보안 이슈**로 지적됐고, 이번 문서에서 이걸 어떻게 고칠지 정리합니다.

---

## 1. 문제 상세 — 왜 위험한가

### 1-1. httpOnly가 원래 막으려던 것

`accessToken` 쿠키를 `httpOnly: true`로 설정해둔 이유는, 만약 사이트 어딘가에 **XSS(악성 스크립트 주입) 취약점**이 하나라도 생기면, 그 악성 스크립트가 `document.cookie`로 세션 토큰을 훔쳐가는 걸 막기 위해서입니다. XSS는 어느 사이트든 실수로 한 번쯤 날 수 있는 흔한 버그 유형이라, 이 방어선은 "혹시 XSS가 터지더라도 세션 탈취까지는 안 되게" 막아주는 마지막 안전장치입니다.

### 1-2. `/api/ws-token`이 이 방어선을 무력화시킴

`GET /api/ws-token`은 **같은 출처(same-origin)에서 오는 요청이면 누가 호출하든 다 응답**해줍니다. 정상적인 프론트 코드가 부르는 것도, 만약 사이트 어딘가에 XSS 취약점이 있어서 공격자 스크립트가 페이지 안에서 실행되고 있는 상황이라면, 그 악성 스크립트도 똑같이:

```js
fetch('/api/ws-token').then(r => r.json()).then(data => {
  // data.accessToken을 공격자 서버로 전송
});
```

이렇게 호출해서, **원래 httpOnly 덕분에 못 훔쳤어야 할 진짜 세션 토큰을 그대로 받아갈 수 있습니다.** 이 토큰은 웹소켓뿐 아니라 일반 REST API 호출에도 그대로 쓸 수 있는, 로그인 세션 전체 권한을 가진 토큰이라 — 사실상 계정 완전 탈취와 같은 피해로 이어집니다. 게다가 이 토큰은 (로그인 응답의 `accessTokenExpiresIn` 기준으로) 수 시간 단위로 유효하기 때문에, 한 번 털리면 피해 노출 시간도 깁니다.

즉 지금 상태는 "**웹소켓 기능 하나를 붙이려고, 사이트 전체의 세션 보안 수준을 낮춰버린**" 상황입니다.

## 2. 요청하는 백엔드 작업

### 2-1. 핵심 아이디어

웹소켓 인증에는 원래의 긴 수명·전체 권한을 가진 `accessToken`을 그대로 쓰지 말고, **웹소켓 연결에만 쓸 수 있는, 아주 짧게 만료되는 별도 "티켓" 토큰**을 새로 발급해주세요. 이러면 설령 이 티켓이 XSS로 털리더라도:
- 몇십 초 안에 만료돼서 공격 가능 시간이 극히 짧고
- 웹소켓 연결 외의 다른 API에는 못 쓰이니 피해 범위가 제한됩니다

### 2-2. 새 엔드포인트 제안

```
POST /auth/ws-ticket
```
- **인증**: 기존 `accessToken`을 `Authorization: Bearer {accessToken}` 헤더로 받아서, 정상 로그인된 사용자인지 확인 (기존 API 인증 방식과 동일)
- **응답 예시**:
```json
{
  "wsTicket": "eyJhbGciOi...(짧은 토큰)",
  "expiresIn": 60
}
```
- **TTL**: 30~60초 정도 제안. 너무 짧으면 네트워크 지연 시 연결 실패가 잦아지고, 너무 길면 보안 이점이 줄어듭니다
- **선택 사항(더 안전하게 하려면)**: 이 티켓을 **1회용**으로 만들어서, 웹소켓 핸드셰이크에서 한 번 성공적으로 쓰이면 즉시 무효화. TTL만으로도 큰 개선이지만, 1회용까지 더하면 훔쳐가도 이미 정상 연결이 끝난 티켓이라 재사용 자체가 불가능해집니다
- 토큰 내용물은 최소한으로 — 사용자 식별 정도만 담으면 되고, 기존 `accessToken`처럼 role 등 다른 클레임까지 담을 필요는 없습니다 (웹소켓 인증 목적 외에는 안 쓰이므로)

### 2-3. WebSocket 핸드셰이크 쪽 반영 필요사항

지금 `ws-coffeechat`의 STOMP CONNECT 인증은 기존 `accessToken`(일반 JWT)을 검증하는 로직으로 되어 있을 겁니다. 이걸 **새로 발급하는 `wsTicket`도 인식해서 검증**하도록 반영이 필요합니다. 두 가지 방식 중 편한 대로 선택하시면 됩니다:

- (A) 기존 accessToken 검증 로직과 별개로, `wsTicket` 전용 검증 로직을 추가해서 CONNECT 시 두 종류의 토큰을 각각 다르게 처리
- (B) `wsTicket`도 기존 JWT와 같은 서명 방식으로 발급하되, 페이로드에 `purpose: "ws"`와 짧은 `exp`만 다르게 넣어서, 기존 검증 로직에서 이 클레임을 보고 분기 처리

기존에 일반 `accessToken`으로 직접 연결하던 경로를 완전히 막을지, 당분간 같이 열어둘지는 백엔드에서 판단해서 진행해주시면 됩니다 (완전히 막는 쪽이 더 안전하지만, 프론트 배포 타이밍과 맞춰야 해서 팀 상황에 맞게 결정).

---

## 3. 이 작업이 끝나면 프론트에서 할 일 (참고용)

백엔드 작업이 끝나면 프론트는 아래 두 곳만 수정하면 됩니다. 코드가 이미 이 구조에 맞게 짜여 있어서 큰 변경은 아닙니다.

**`src/app/api/ws-token/route.ts`** — 지금은 쿠키의 accessToken을 그대로 반환하는데, 이 대신 서버사이드에서 `POST /auth/ws-ticket`을 호출해서 그 결과(`wsTicket`)를 반환하도록 교체.

**`src/features/user/coffee-chat/hooks/useCoffeeChatSocket.ts`** — 사실상 수정 거의 불필요. 이미 "연결하거나 재연결할 때마다 매번 `/api/ws-token`을 새로 호출해서 최신 토큰을 받아오는" 구조라, 짧게 만료되는 티켓 방식과 자연스럽게 맞물립니다.

---

## 4. 테스트 방법

1. `POST /auth/ws-ticket` 호출 시 30~60초 정도 후 만료되는 짧은 토큰이 오는지 확인
2. 이 토큰으로 `ws-coffeechat` STOMP CONNECT가 정상적으로 성공하는지 확인
3. 발급 후 TTL이 지난 토큰으로 CONNECT 시도 시 정상적으로 거부(ERROR 프레임 + close 1002)되는지 확인
4. (1회용으로 구현했다면) 같은 티켓으로 두 번째 CONNECT 시도 시 거부되는지 확인
5. 기존 일반 `accessToken`으로 REST API 호출은 지금처럼 정상 동작하는지 확인 (이번 변경이 다른 API 인증에 영향 없어야 함)
